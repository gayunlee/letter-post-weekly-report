# 분류 체계 방향 정리: 4축 → 2축 + detail_tags 강화

날짜: 2026-02-22

## 배경

VOC 분류 체계 고도화 OpenSpec에서 4축(Topic × Sentiment × Intent × Urgency) + 부서 라우팅을 계획했으나, 실제 부서 활용 시나리오를 검토하면서 방향을 재정리함.

## 결정

1. **별도 축 추가하지 않음**: Intent(의도), Urgency(긴급도), Department(담당부서)를 분류 축으로 추가하지 않는다.
2. **기존 2축 + detail_tags 유지**: Topic(4) × Sentiment(3) + 카테고리 태그(28) + 자유 태그 + 요약이 충분한 분석 자유도를 제공한다.
3. **Intent는 부수 추출**: detail_tags 추출 시 동일 Haiku 호출에서 Intent도 함께 뽑되, 별도 축이 아닌 쿼리용 보조 필드로만 활용한다 (추가 비용 0).
4. **자유도 우선 설계**: 특정 부서의 뷰를 하드코딩하지 않고, 부서마다 자기 목적에 맞게 조합·쿼리할 수 있게 한다.

## 근거

- **담당부서 라우팅**: 부서마다 같은 데이터를 다른 목적으로 본다. CS팀은 "서비스이슈+부정+결제태그"로 필터하고, 사업팀은 "마스터별 감성 추이"로 보고, 콘텐츠팀은 "콘텐츠반응+부정"으로 본다. 이걸 미리 분류할 필요가 없다.
- **긴급도**: "긴급한 건"의 정의가 부서마다 다르다. CS에게 긴급한 건(결제 오류)과 콘텐츠팀에게 긴급한 건(마스터 불만 급증)이 다르다.
- **Intent**: Topic+Sentiment+카테고리태그 조합으로 대부분의 의도 파악이 가능. "서비스이슈+중립+결제/환불" = 결제 문의, "콘텐츠반응+부정+콘텐츠품질" = 콘텐츠 불만 피드백.

## 검증 필요 사항

기존 2축 + detail_tags 체계로 아래 시나리오가 모두 대응되는지 확인:
- CS: 결제/로그인 관련 즉시 대응 건 추출
- 사업: 마스터별 긍부정 여론 추적
- 콘텐츠: 콘텐츠 불만 패턴 분석
- 전사: 상품 런칭 후 반응 모니터링, 증시 상황 vs 감정 상관관계

→ `openspec/specs/voc-classification/analysis-scenarios.md`에 7개 시나리오 작성 완료
