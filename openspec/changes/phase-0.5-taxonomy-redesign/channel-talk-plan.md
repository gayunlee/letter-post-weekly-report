# 채널톡 CS 데이터 2축 분류 설계 (확정)

## 데이터 탐색 결과

- **소스**: BigQuery `us-service-data.channel_io.messages` + `channel_io.chats`
- **기간**: 2024-08-06 ~ 2025-12-05 (약 16개월)
- **총 메시지**: 431,589건, 주간 ~1,000건 대화
- **personType**: bot 68% / user 20% / manager 12%
- **중복**: 모든 메시지가 2번씩 존재 → dedup 필수
- **state** (chats): closed 21,154 / opened 14,228 (40% 방치)
- **워크플로우**: 봇 인사 → 버튼 선택 → 봇 가이드 → (1:1 전환) → 매니저 응답 → 종료

## 분류 목적

1. 서비스/콘텐츠 개선 인사이트
2. CS 자동화 기회 식별
3. 환불/강의 런칭 후 문의 패턴 분석
4. 날짜·마스터별 시계열 추적

## 2축 분류 체계

### Axis 1: 해결 경로 (route) — 규칙 기반

| route | 조건 | 설명 |
|-------|------|------|
| `manager_resolved` | manager_message_count > 0 AND state="closed" | 매니저 개입 해결 |
| `bot_resolved` | manager_message_count == 0 AND state="closed" | 봇만으로 해결 (워크플로우) |
| `abandoned` | state="opened" OR (state="closed" AND 사용자 메시지 없음) | 이탈/방치/무응답 |

판정 로직:
```python
def detect_route(chat_item, chat_state):
    if chat_state == "closed":
        if chat_item["manager_message_count"] > 0:
            return "manager_resolved"
        else:
            return "bot_resolved"
    else:  # opened
        return "abandoned"
```

### Axis 2: 문의 주제 (topics) — 다중 태그 LLM 분류

**다중 태그 방식**: 관련된 주제를 모두 태그. 단일 선택이 아닌 multi-label.

| topic | 설명 | 신호 |
|-------|------|------|
| 결제·환불 | 결제, 취소, 환불, 계좌이체 | "환불", "결제", "취소" |
| 구독·멤버십 | 구독, 해지, 변경, 갱신, 등급 | "해지", "구독", "멤버십" |
| 콘텐츠·수강 | 강의 접근, 자료, 수강, 오프라인 행사 | "강의", "수강", "자료", "세미나" |
| 기술·오류 | 로그인, 앱 오류, 접속 문제 | "오류", "로그인", "안됨" |
| 기타 | 위에 해당 없음 | 단순 인사, 기타 |

### 다중 태그 처리 (v2 — 2026-03-13 전환)

이전 방식 (v1): 단일 topic + 우선순위 규칙 → 결제↔구독 경계에서 오분류 빈발
현재 방식 (v2): **관련된 주제를 모두 topics 배열에 포함**

- "구독 해지 + 환불해주세요" → `["결제·환불", "구독·멤버십"]`
- "로그인 안 됨 + 환불" → `["기술·오류", "결제·환불"]`
- "강의 어디서 보나요" → `["콘텐츠·수강"]` (단일)

장점:
1. 우선순위 규칙 불필요 → 프롬프트 단순화
2. 결제↔구독 경계 문제 해소 (50건이 둘 다 태그)
3. 교차 분석 가능 (결제+구독 동시 = 해지환불 패턴)
4. 중복 카운트 허용 — "결제·환불 관련 문의 N건"에 다중 태그 건도 포함

### 분류 대상

- **topics 분류**: `manager_resolved` + `bot_resolved`만 (LLM API 호출)
- **abandoned**: route만 기록, topics = `[]` (분류 의미 없음)

### 워크플로우 버튼 활용

사용자 메시지 중 알려진 워크플로우 버튼 텍스트를 파싱하여 topic 힌트로 사용:
- "구독해지 및 환불" → 결제·환불, 구독·멤버십
- "사이트 및 동영상 오류" → 기술·오류
- "수강 및 상품문의" → 콘텐츠·수강
- "기타 문의" → LLM에 넘김

버튼 텍스트는 LLM 프롬프트에 `[워크플로우: {버튼}]` 접두사로 포함.

## 분류 규칙

### 결제·환불
결제, 환불, 계좌이체, 카드 결제/변경, 영수증 요청, 의도치 않은 자동결제.

### 구독·멤버십
구독 신청, 해지, 등급 변경, 갱신, 멤버십 관련. 미이용 강의 신청 의사도 포함.

### 콘텐츠·수강
이미 이용 중인 강의 접근, 수강 방법, 자료/교재 요청, 이용 방법. **오프라인 행사/세미나/특강도 여기**.

### 기술·오류
확실히 시스템 문제인 경우. 불확실하면 콘텐츠·수강. 기술 기능 건의도 포함.

### 기타
위 4개에 해당하지 않는 문의. 단순 인사, 투자 질문, 비기술 기능 개선 건의.

## 현재 아키텍처 (확정, 2026-03-15)

### 1차: KcELECTRA 3분류 (라우팅)

- 모델: `beomi/KcELECTRA-base-v2022` (110M)
- 분류: 결제·구독 / 콘텐츠·수강 / 기술·오류
- 분류 기준: **"발생한 현상 / 고객이 요청한 사항"** (의도가 아닌 현상 기준)
- 학습: Vertex AI T4, 3,502건 (Opus + Claude 직접 라벨링)
- 추론: 로컬 ~10ms/건
- conf ≥ 0.9: KcELECTRA 확정 (93% of items)
- conf < 0.9: LLM fallback (7% of items)

### 2차: LLM 서브태그 (인사이트)

- 모델: Haiku (또는 Claude Code)
- 역할: 1차 분류 결과 내에서 서브태그 부여
- 서브태그 예시:
  - 결제·구독 → [환불/해지/상품변경/카드변경/자동결제항의/신규가입]
  - 콘텐츠·수강 → [수강방법/오프라인참석/녹화본/자료요청/동반참석/불참통보]
  - 기술·오류 → [로그인불가/앱오류/콘텐츠접근차단/기능건의/결제연동]
- 서브태그가 다른 1차 클래스 키워드를 포함 가능
  - 기술·오류 → 서브태그 `결제연동` (결제했는데 멤버십 안올라옴)
  - 결제·구독 → 서브태그 `시스템장애` (해지했는데 자동결제 또 됨)

### 파이프라인

```
BigQuery → dedup → build_chat_items → route 판정
  → 1차: KcELECTRA 3분류 (로컬, 10ms/건)
    → conf ≥ 0.9: 확정
    → conf < 0.9: LLM fallback
  → 2차: LLM 서브태그 (API)
  → 저장 + 통계 + 주간 리포트
```
