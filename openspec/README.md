# OpenSpec 구조

이 디렉토리는 프로젝트의 스펙, 변경 제안, 의사결정 기록을 관리합니다.

## 디렉토리 구조

```
openspec/
├── README.md                  ← 이 파일
├── specs/                     ← 현행 스펙 (현재 시스템이 어떻게 동작하는지)
│   ├── voc-classification/
│   │   ├── spec.md            ← VOC 분류 시스템 스펙
│   │   └── analysis-scenarios.md  ← 부서별 데이터 분석 시나리오
│   └── data-sync/
│       └── spec.md            ← 데이터 동기화 스펙
├── changes/                   ← 변경 제안 (미구현 또는 진행중인 변경)
│   ├── phase-0-four-axis-schema/  ← 예: Phase 0 변경 제안
│   │   ├── proposal.md        ← 왜 이 변경이 필요한가
│   │   ├── design.md          ← 기술적 접근 방법
│   │   ├── tasks.md           ← 구현 체크리스트
│   │   └── specs/.../spec.md  ← 관련 스펙의 delta (추가/변경/삭제)
│   └── phase-1-review-pipeline/
│       └── ...
└── decisions/                 ← 의사결정 및 벤치마크 기록 (시간순)
    └── YYYY-MM-DD-제목.md     ← 각 기록은 날짜 prefix로 정렬
```

## decisions/ 사용법

주요 의사결정, 벤치마크 결과, 방향 전환 등을 시간순으로 기록합니다.

### 기록 시점
- 모델 벤치마크 결과가 나왔을 때
- 분류 체계나 태그 목록을 변경할 때
- 기술 스택 선택/변경 시
- 검수 결과로 방향이 바뀔 때
- A/B 테스트 결과가 나왔을 때

### 파일명 규칙
```
YYYY-MM-DD-간단한-제목.md
예: 2026-02-22-intent-축-제거-결정.md
    2026-03-01-topic-재학습-결과.md
    2026-03-15-채팅-인터페이스-poc-결과.md
```

### 템플릿
```markdown
# 제목

날짜: YYYY-MM-DD

## 배경
어떤 상황에서 이 결정/실험이 필요했는지

## 결과 (벤치마크인 경우)
수치, 비교 테이블 등

## 결정
무엇을 결정했는지

## 근거
왜 이렇게 결정했는지

## 다음 단계
이 결정에 따른 후속 작업
```

## specs/ vs changes/ 차이

- **specs/**: 현재 시스템의 "있는 그대로" 스펙. 구현이 완료되면 여기에 반영.
- **changes/**: 아직 구현되지 않았거나 진행중인 변경 제안. 완료되면 specs/에 병합.
